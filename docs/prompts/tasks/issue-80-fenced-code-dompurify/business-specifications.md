# Business Specifications — Issue #80: Fenced Code in Markdown Renders Complex HTML

## Goal and Scope

Issue #75 extended the introduction extractor to capture `<pre>` elements from the blog HTML. However, fenced code blocks in blog articles are not rendered as bare `<pre>` elements. They are rendered as rich, nested HTML structures: a wrapping `<div class="highlight">` containing a `<div class="chroma">`, a `<table class="lntable">` with line-number cells and code cells each containing `<pre>` elements, and a `<button class="copyCodeButton">`. This structure contains elements and attributes — most notably `<table>`, `<tbody>`, `<tr>`, `<td>`, and `<button>` — that DOMPurify strips by default.

As a result, when the user copies the Body HTML for Medium or Substack and pastes it into those platforms, fenced code blocks are rendered incorrectly or not at all: the line-number table, code highlighting structure, and copy button are stripped or corrupted.

The goal of this change is twofold:

1. Teach DOMPurify, which is already in use on `PlatformMedium.vue` and `PlatformSubstack.vue`, to allow the full HTML structure that a fenced code block produces so that copying and pasting the Body HTML into Medium or Substack preserves the code block as intended.
2. Provide a full-article HTML test fixture — saved as a file in the project — sourced from the live article https://iamjeremie.me/post/2026-03/organizing-specifications-with-claude-code/ so that fenced code block rendering is covered by a non-regression test.

X and LinkedIn are not affected; their content generators do not produce or render HTML through `v-html`, so no DOMPurify configuration applies to them.

## Affected Files

### `src/components/platforms/PlatformMedium.vue`

This component sanitizes the `rawBodyHtml` value using DOMPurify before binding it to `v-html` for live preview, and before writing it to the clipboard. The DOMPurify call must be reconfigured to allow the HTML elements and attributes that fenced code block structures rely on, while still blocking all active content (scripts, event handlers, `javascript:` URLs, etc.).

### `src/components/platforms/PlatformSubstack.vue`

Same role and same requirement as `PlatformMedium.vue`. Both components use the identical DOMPurify sanitization pattern; both must be updated consistently.

### Test fixture file — full article HTML

A new HTML fixture file containing the full fetched HTML of the live article https://iamjeremie.me/post/2026-03/organizing-specifications-with-claude-code/ must be saved under `tests/fixtures/`. This file is the authoritative source for what a fenced code block looks like in the blog's rendered output, and it drives the non-regression test for introduction extraction of fenced code content.

### Existing test files that cover the affected code paths

The unit tests for `htmlExtractor.ts` and for the Medium and Substack content generators must be extended (not replaced) to use the new fixture and assert that the fenced code block structure survives extraction and sanitization intact.

## Rules and Constraints

1. DOMPurify must continue to strip all active content: `<script>` tags, inline event handlers (e.g. `onclick`, `onerror`), `javascript:` URLs, `<iframe>`, `<object>`, `<embed>`, and any other constructs that can execute code in the browser. This constraint is non-negotiable.
2. The expanded DOMPurify allowlist must include at minimum the elements and attributes present in the fenced code block HTML structure produced by the blog engine, as observed in the live article. No element or attribute that is not required by the known output structure should be added gratuitously.
3. The `<button class="copyCodeButton">` element generated by the blog's fenced code renderer is part of the extracted HTML. The spec does not mandate that this button must remain interactive or functional when pasted into Medium or Substack; the requirement is only that DOMPurify does not silently remove the structural HTML that surrounds and contains the code content.
4. The DOMPurify configuration must be consistent between `PlatformMedium.vue` and `PlatformSubstack.vue`. Diverging configurations between the two components are not permitted.
5. The new fixture file must be the actual HTML fetched from the live article, not a hand-crafted approximation. It is used as a non-regression test so it must reflect the real structure.
6. The non-regression test must assert that, after extraction and sanitization, the text content of the fenced code block is present in the output. A test that only checks the absence of errors is not sufficient.
7. The clipboard copy behaviour (existing `copyRenderedHtml` logic in both components) copies the sanitized HTML. After this fix the sanitized HTML must contain the fenced code block content; it must not be empty where fenced code previously appeared.

## Example Mapping

### Rule: The DOMPurify allowlist is extended to preserve fenced code block structure

**Example 1 — Fenced code block survives sanitization**
Given a `rawBodyHtml` that contains a fenced code block rendered as `<div class="highlight"><div class="chroma">...<pre>...</pre>...</div></div>`,
when DOMPurify sanitizes that HTML,
then the resulting sanitized HTML contains the code text that was inside the `<pre>` elements, and the outer structural elements (`div.highlight`, `div.chroma`, the table elements, and `pre`) are preserved.

**Example 2 — Active content is still stripped after the allowlist is extended**
Given a `rawBodyHtml` that contains both a fenced code block and an injected `<script>alert('xss')</script>`,
when DOMPurify sanitizes that HTML,
then the `<script>` element is absent from the sanitized output, and the fenced code block content is still present.

**Example 3 — Inline event handler is still stripped**
Given a `rawBodyHtml` that contains `<pre onclick="stealCookies()">code</pre>`,
when DOMPurify sanitizes that HTML,
then the `onclick` attribute is absent from the sanitized output, and the `<pre>` element with its text content is still present.

**Example 4 — Medium preview renders fenced code block**
Given an article whose introduction contains a fenced code block,
when the user navigates to the Medium platform page,
then the live Body HTML preview (the `v-html` div) renders the code block with its content visible, not as a blank region or a broken table.

**Example 5 — Substack preview renders fenced code block**
Given an article whose introduction contains a fenced code block,
when the user navigates to the Substack platform page,
then the live Body HTML preview renders the code block with its content visible, identically to what Medium shows.

**Example 6 — Copying the Body HTML includes fenced code block content**
Given an article whose introduction contains a fenced code block,
when the user clicks "Copy" on the Body HTML field on either the Medium or Substack page,
then the clipboard contains HTML in which the code text from the fenced block is present and the structural wrapper elements are intact.

**Example 7 — Article with no fenced code block is unaffected**
Given an article whose introduction contains only `<p>` elements and no fenced code,
when DOMPurify sanitizes the Body HTML,
then the sanitized output is identical to what was produced before this fix.

**Example 8 — Non-regression test using the full fixture**
Given the full HTML of the article https://iamjeremie.me/post/2026-03/organizing-specifications-with-claude-code/ saved as a fixture,
when the introduction is extracted and the resulting Body HTML is sanitized,
then the code text from the fenced block in that article's introduction is present in the sanitized output.

## Edge Cases (Observable Consequences)

- A fenced code block whose inner code contains HTML-like text (e.g. a `<div>` written as literal text in the code block) must have that text rendered as plain text inside `<code>` or `<pre>`, not interpreted as HTML. If the blog engine encodes such text as HTML entities inside `<span>` elements, the sanitized output must preserve those entities unchanged so the rendered text shows the correct characters.
- A `<button class="copyCodeButton">` inside the fenced code wrapper that contains an `onclick` or similar event handler must have that handler stripped by DOMPurify. The button element itself may or may not survive sanitization; the requirement is only that no event handler survives.
- An article introduction that contains multiple fenced code blocks must have all of them preserved after sanitization, not just the first.
- An article introduction that mixes fenced code blocks with `<p>`, `<ul>`, and `<blockquote>` elements (all of which were added in issue #75) must have all element types preserved after sanitization; the extended allowlist must not accidentally block elements that DOMPurify previously allowed by default.
- The `v-html` preview in `PlatformMedium.vue` and `PlatformSubstack.vue` renders inside a `prose` Tailwind class container. The visual appearance of the fenced code block in that preview is not within the scope of this fix; only the presence of the code content is required.

## Concurrency and Performance

The DOMPurify sanitization already runs synchronously inside a `computed` property in both platform components. This fix modifies only the configuration passed to DOMPurify; it does not change the synchronous nature of the operation. No asynchronous processing is introduced. The sanitized output must be available in the same render cycle as before.
