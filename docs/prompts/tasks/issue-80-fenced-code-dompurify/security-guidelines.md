# Security Guidelines — Issue #80: Fenced Code DOMPurify Allowlist Extension

## Scope

These rules apply to the DOMPurify configuration change in `PlatformMedium.vue` and `PlatformSubstack.vue`, the HTML fixture added under `tests/fixtures/`, and the test extensions that exercise the sanitization path.

---

## Rules

### 1. Active content must remain blocked regardless of allowlist expansion

**What:** The DOMPurify configuration must continue to block, at minimum: `<script>`, `<iframe>`, `<object>`, `<embed>`, `<form>`, inline event handler attributes (`onclick`, `onerror`, `onload`, and all `on*` variants), and `javascript:` URI schemes. No element or attribute that can execute code in the browser may appear in the final allowlist.

**Where:** `src/components/platforms/PlatformMedium.vue` and `src/components/platforms/PlatformSubstack.vue` — the `DOMPurify.sanitize()` call and its configuration object.

**Why:** DOMPurify's default configuration already blocks all active content. Switching from the default call to an explicit `ALLOWED_TAGS` or `ADD_TAGS` configuration introduces the risk of inadvertently re-enabling dangerous tags that were previously blocked implicitly. The fix must be additive (permitting structural tags only) and must not weaken the active-content barrier that was already established by ADR-007.

---

### 2. The allowlist must be the minimal set required by the observed fenced-code structure

**What:** Only elements and attributes that are demonstrably present in the blog engine's rendered fenced-code-block HTML output may be added to the DOMPurify allowlist. No speculative additions are permitted. Each added element or attribute must be traceable to the live article fixture or the business specification.

**Where:** `src/components/platforms/PlatformMedium.vue` and `src/components/platforms/PlatformSubstack.vue` — the DOMPurify configuration.

**Why:** An overly broad allowlist enlarges the XSS attack surface. If a user pastes attacker-controlled HTML into the Body HTML textarea, a permissive allowlist increases the chance that a crafted payload survives sanitization and is rendered via `v-html`. The allowlist must be scoped to the known output structure, not to everything DOMPurify supports.

---

### 3. The `class` attribute, if permitted, must be understood as a presentation risk, not an execution risk, and must not be used as an execution vector

**What:** The fenced-code block structure uses `class` attributes (`highlight`, `chroma`, `lntable`, `copyCodeButton`, etc.). If the allowlist permits `class`, no `class` value can be treated as trusted by application logic or used to trigger JavaScript behaviour. DOMPurify must not be configured to allow `style` attributes or `<style>` tags alongside the class allowance.

**Where:** `src/components/platforms/PlatformMedium.vue` and `src/components/platforms/PlatformSubstack.vue` — DOMPurify configuration; the `v-html` preview div.

**Why:** `class` attributes are not execution vectors on their own, but `style` attributes and `<style>` blocks can be used for CSS-injection attacks (e.g., expression() in legacy browsers, or data exfiltration via CSS selectors). The `class` allowance must not be expanded to cover `style`.

---

### 4. The `<button>` element, if preserved, must have all event handlers stripped

**What:** The `<button class="copyCodeButton">` element generated by the blog engine may survive sanitization as a structural element. Any `onclick`, `onmousedown`, or other event handler on that button must be stripped by DOMPurify before the HTML is bound to `v-html` or written to the clipboard.

**Where:** `src/components/platforms/PlatformMedium.vue` and `src/components/platforms/PlatformSubstack.vue` — the DOMPurify sanitization step.

**Why:** A `<button>` element is not inherently dangerous, but a button with an `onclick` handler rendered via `v-html` would execute that handler in the context of the application. The business specification explicitly states that the button's interactivity is not required; the constraint is therefore that the element may survive but its handlers must not.

---

### 5. Both platform components must use an identical DOMPurify configuration

**What:** The DOMPurify configuration (tags, attributes, and any additional options) must be identical between `PlatformMedium.vue` and `PlatformSubstack.vue`. The configuration must be defined in a single shared location (a utility or constant), not duplicated inline in each component.

**Where:** A new shared utility (e.g., under `src/utils/`) that both platform components import; or a composable that encapsulates the sanitization call.

**Why:** Diverging configurations between the two components create a class of vulnerability where a security fix applied to one component is silently absent from the other. A single definition eliminates the possibility of configuration drift. This is a defence-in-depth requirement; it also mirrors the business constraint that both components must be consistent.

---

### 6. The HTML fixture file must not be served or executed as live HTML

**What:** The fixture file saved under `tests/fixtures/` must be treated as a static data file consumed only by test code. It must not be importable as a route, renderable by the dev server as a page, or processed by Vite as an entry point.

**Where:** `tests/fixtures/` — file placement and test infrastructure configuration.

**Why:** The fixture is the raw, unsanitized HTML of a live article. It contains the full DOM of an external page, which may include `<script>` tags and other active content (expected to be stripped by DOMPurify in the test, but not sanitized at rest). If the file were accidentally served and rendered by a browser, it would execute that content in the application's origin.

---

### 7. Sanitization must occur before both `v-html` binding and clipboard write — never only before one

**What:** The `sanitizedBodyHtml` computed value must be the single source used for both the `v-html` preview binding and the `copyRenderedHtml` clipboard write. Neither path may use `rawBodyHtml` directly.

**Where:** `src/components/platforms/PlatformMedium.vue` and `src/components/platforms/PlatformSubstack.vue` — the `copyRenderedHtml` function and the `v-html` binding.

**Why:** If the clipboard write path were changed to copy `rawBodyHtml` instead of `sanitizedBodyHtml` (for example, to preserve formatting), unsanitized user-edited content — potentially containing `<script>` tags or event handlers — would be placed on the clipboard. A target application that subsequently renders that clipboard content would be exposed to the injected payload.

---

### 8. The Netlify Function domain allowlist is unaffected by this change and must not be modified

**What:** The `ALLOWED_DOMAINS` array in `netlify/functions/fetch-article.ts` must remain unchanged. This fix does not expand the set of domains from which articles can be fetched.

**Where:** `netlify/functions/fetch-article.ts`.

**Why:** The domain allowlist is the primary server-side boundary preventing the proxy function from being used to fetch arbitrary external content. The fenced-code fix operates entirely on already-fetched HTML in the client; it does not require any change to which origins the proxy will contact. Broadening the allowlist without a corresponding requirement would expand the SSRF surface of the proxy function.

---

### ADR Required

**New security pattern: Shared DOMPurify configuration module**

The existing ADR-007 documents the decision to use DOMPurify and describes it as a call with default configuration. Issue #80 requires switching from the default call to an explicit, non-default configuration (an expanded allowlist). This introduces a new pattern: a shared sanitization configuration that is imported by multiple components rather than being inline in each.

ADR-007 should be updated, or a successor ADR created, to document:
- The decision to define the DOMPurify configuration in a shared location rather than inline per-component.
- The governance rule that the allowlist may only be extended when a concrete structural requirement from an observed blog output is present, not speculatively.
- The constraint that any future DOMPurify allowlist change must be accompanied by a corresponding non-regression test that exercises the newly permitted structure and asserts that active content is still blocked.

This pattern is not currently documented in `docs/decisions/` and establishes a precedent for how DOMPurify configuration is managed as the allowlist evolves over time.
